## 1. Continuité v0.9.0 → v0.9.1

La v0.9.0 a livré, côté MCP : 19 outils fonctionnels au démarrage de Phase 3 puis le port archeo natif Phases 0/2 + Phase 3 niveau `tags` + orchestrator + 9ᵉ catégorie health-scan `mcp-tool-spec-drift`, avec **189 tests** et **84 % coverage**.

La v0.9.1 boucle :

- **Phase 3 niveaux non-tags** : `releases`, `merges`, `commits` + mode `--branch-first`. Le seul vrai vide fonctionnel restant côté MCP est comblé.
- **Doctrine linking** posée en v0.7.4 mais jamais mécaniquement enforced est désormais appliquée en écriture par `mem_archive`.
- **10ᵉ catégorie health-scan** ferme le risque de re-livrer un SKILL.md > 1024 chars (bug fixé en v0.7.5, désormais sous filet).

À l'arrivée : **224 tests pytest, 83 % coverage**. La baisse d'1 point de coverage est mécanique — ~600 lignes ajoutées dans `archeo_git.py` dont les chemins GH-mocked ne sont pas exhaustivement testés (tests E2E vrais GH renvoyés v1.0).

## 2. Décisions structurantes (3 axes)

### Axe 1 — Cohérence des paramètres et de l'idempotence à travers les 4 niveaux

Le défi du port Phase 3 complet : 4 niveaux (`tags`, `releases`, `merges`, `commits`) qui ont des modèles de données différents (un tag = (name, sha), une release = (tag, body, url), une PR = (number, base, head, mergeCommit), une fenêtre = (label, range, commits)) doivent partager **une seule API** avec **une seule sémantique d'idempotence**.

Solution : un `MilestoneInfo` Pydantic unifié qui porte les champs propres à chaque kind, plus une fonction `_milestone_id(info)` qui produit la clé d'idempotence canonique :

| kind     | source_milestone   | filename suffix          |
|----------|--------------------|--------------------------|
| `tag`    | `v0.8.0`           | `archeo-git-v0-8-0`      |
| `release`| `release-v0.8.0`   | `archeo-git-release-v0-8-0` |
| `merge`  | `pr-#42`           | `archeo-git-merge-pr-42` |
| `window` | `window-2026-W18`  | `archeo-git-window-2026-w18` |

Les filenames préfixés par `{date}-{time}-{slug}-` restent triables chronologiquement dans Obsidian, et le préfixe par kind permet à un `glob` de filtrer rapidement par niveau.

### Axe 2 — Pas de nouvelle dépendance Python pour GitHub

Pour les niveaux `releases` et `merges`, l'accès aux métadonnées GitHub est nécessaire (release notes, PR titles/bodies, base/head branches). Trois options évaluées :

| Option | Pour | Contre |
|---|---|---|
| `gh` CLI subprocess | Pattern existant déjà avec `git`. Auth gérée par gh. Marche avec GH Enterprise. Pas de dep nouvelle. | Suppose `gh` installé. |
| `httpx` raw API calls | Aucune dep en plus de Python core. | Auth manuelle, rate limiting manuel, URLs Enterprise manuelles. |
| `PyGithub` | Lib mature, abstrait l'API. | Grosse dep, couplage fort. |

**Choix : `gh` CLI subprocess.** Pattern cohérent avec `git`, zéro dep Python ajoutée. Si `gh` est absent → le niveau émet un warning explicite et retourne une liste vide ; le LLM client peut basculer sur le skill fallback. Comportement testé (`test_releases_skips_when_gh_unavailable`).

### Axe 3 — Branch-first comme paramètre, pas comme outil séparé

Plutôt que de créer un outil `mem_archeo_branch` distinct, le mode branch-first devient un paramètre `branch_first: str | None` sur `mem_archeo_git` ET `mem_archeo`. Quand défini, exige aussi `branch_base: str` (défaut `"main"`). Active un dispatcher dédié `_discover_branch_first` qui :

1. Résout la divergence via `git merge-base {base} {branch}`.
2. Pour `level='commits'` (défaut implicite en branch-first) : enumerate `git log --no-merges {base_sha}..{branch}` et délègue à `_list_commit_windows` avec `rev_range` pré-calculé.
3. Pour `level='merges'` : `git log --merges {base_sha}..{branch}`, 1 archive par merge commit unique à la branche.
4. Pour `level='tags'` ou `level='releases'` (signaux repo-wide, peu de sens en branch-first) : warning + fallback sur commits-by-author.

Granularité par défaut `--by-author` quand branch-first (pattern v0.7.1). Le frontmatter de chaque archive porte `branch`, `branch_base`, `branch_base_sha` (vides hors branch-first), permettant à `mem-archeo` orchestration de wirer une topologie de branche dédiée dans `99-meta/repo-topology/{slug}-branches/` (déférée à v1.0 — pour l'instant les archives portent l'info, la topologie n'est pas écrite).

## 3. Bloc 1 — Port archeo Phase 3 complet

### 3.1 Architecture

`tools/archeo_git.py` passe d'un module mono-niveau (~620 lignes en v0.9.0) à un dispatcher (~1100 lignes en v0.9.1). Découpage interne :

```
execute_git(...)
├── _discover_milestones(level, ...) — dispatcher
│   ├── _list_semver_tags(...)        — niveau tags (v0.9.0)
│   ├── _list_github_releases(...)    — niveau releases (NEW)
│   ├── _list_github_merges(...)      — niveau merges (NEW)
│   └── _list_commit_windows(...)     — niveau commits (NEW)
├── _resolve_branch_context(...)      — branch-first divergence (NEW)
├── _discover_branch_first(...)       — branch-first dispatcher (NEW)
├── _commit_metadata(...)             — helper partagé (extrait)
├── _co_authors_for_commits(...)      — extraction trailers (NEW)
├── _window_label(...)                — calcul ISO week / month (NEW)
├── _milestone_id(...)                — clé d'idempotence canonique (NEW)
├── _archive_filename(...)            — filename par kind (refacto)
├── _milestone_headline(...)          — Main section par kind (refacto)
├── _build_body(...)                  — Main + AI files + Friction
└── _write_archive(...)               — write avec frontmatter par kind
```

### 3.2 Modèle `MilestoneInfo` étendu

Ajout d'un champ pivot `milestone_kind: str` (`tag` | `release` | `merge` | `window`) et de groupes de champs par kind, optionnels :

- Release : `release_tag`, `release_url`, `release_is_prerelease`, `release_is_draft`
- Merge : `pr_number`, `pr_url`, `pr_base`, `pr_head`, `pr_merged_at`
- Window : `window_label`, `window_start`, `window_end`, `commit_count`, `co_authors`

Tous les champs de l'ancien modèle (`tag`, `commit_sha`, `date`, `time`, `author_*`, `subject`, `files_changed`, etc.) deviennent défaultés à `""` ou `0` pour rester rétro-compatibles avec le niveau `tags`.

### 3.3 Tests

8 nouveaux tests dans `test_archeo_git.py` (total 18 archeo_git + 5 archeo orchestrator) :

- 3 `commits` : grouping par semaine, split `by_author`, filtrage `since/until`.
- 2 `releases` : créations multiples avec frontmatter checké, skip silencieux quand `gh` absent.
- 1 `merges` : 2 PRs (squash + merge commit) → 2 archives avec frontmatter PR-spécifique.
- 2 `branch_first` : scoping correct sur commits uniques, fallback gracieux quand branche inexistante.

`gh` CLI mocké via `monkeypatch.setattr("memory_kit_mcp.tools.archeo_git.subprocess.run", fake_run)` qui intercepte les commandes `gh` et passe `git` au real subprocess.

## 4. Bloc 2 — Doctrine `_linking.md` mécaniquement enforced

### 4.1 Le gap

L'invariant `_linking.md` (v0.7.4) prescrit : « Every `[[wikilink]]` written into a persisted vault file must resolve to an existing target at the time of writing. » Mais en v0.7.4 → v0.9.0, cet invariant n'était **vérifié qu'a posteriori** par `mem_health_scan` (catégorie `dangling-wikilinks`, severity `info`). Rien n'empêchait l'introduction de nouveaux dangling à l'écriture.

### 4.2 La solution

`mem_archive` valide pre-write :
- Mode incremental : `context_md` body.
- Mode full : `archive_body_md` ET `new_context_md` bodies.

Si dangling détecté → raise `DanglingWikilinkError` (sous-classe `ValueError`) listant les targets, avec message actionnable pointant vers les conventions de bypass (inline backticks `` `[[X]]` ``).

### 4.3 Refactor pour DRY

`strip_code` + `WIKILINK_RE` étaient privés dans `health/scan.py`. Extraction dans un nouveau module `vault/wikilinks.py` consommé par les deux : scanner et enforcement. Garantit que ce que l'un valide, l'autre ne flagge pas (et vice-versa).

API de `vault/wikilinks.py` :

- `WIKILINK_RE` — regex canonique.
- `strip_code(body)` — strip fenced + inline code.
- `find_wikilinks(body)` — extraction de tous les `[[X]]` hors code.
- `build_vault_index(vault)` — `(basenames, relpaths_no_ext)`.
- `resolve_wikilink(target, basenames, relpaths_no_ext)` — booléen.
- `find_dangling(body, vault, exempt=None)` — entry-point combiné, retourne la liste de-duplicated en ordre document.
- `build_incoming_index(vault_files, vault)` — réservé pour un futur refactor de health/scan (orphan-atoms detection).

### 4.4 Risque vault prod

Vérifié lors de la session 2026-05-04 : le vault prod a **0 dangling-wikilinks** (mesure via `mem_health_scan --only dangling-wikilinks`). L'enforcement est safe à l'instant T.

## 5. Bloc 3 — 10ᵉ catégorie health-scan `skill-description-too-long`

### 5.1 Le bug couvert

La limite Anthropic de 1024 chars sur `description` du frontmatter SKILL.md est silencieuse : un dépassement casse le déclenchement automatique du skill sur Claude Code, Codex, Vibe, Copilot, sans message d'erreur. Le bug a été fixé en v0.7.5 (toutes les descriptions ramenées sous le cap), mais aucun garde-fou n'empêchait une régression à l'édition suivante.

### 5.2 La 10ᵉ catégorie

`scan_vault(..., kit_repo=...)` audite désormais récursivement :

- `kit_repo/adapters/claude-code/skills/*.template.md`
- `kit_repo/adapters/codex/skills/*/SKILL.md.template`
- `kit_repo/adapters/mistral-vibe/skills/*/SKILL.md.template`
- `kit_repo/adapters/copilot-cli/skills/*/SKILL.md.template`

Pour chaque fichier match, parse YAML frontmatter, check `len(description) > 1024` → finding `warning`. Skip silencieux si `kit_repo` ou `adapters/` absent. **Gemini exclu** : ses commands utilisent TOML literal strings, pas le format Anthropic SKILL.md, donc pas de cap équivalent.

### 5.3 Mcp-only par design

Comme la 9ᵉ catégorie `mcp-tool-spec-drift`, cette 10ᵉ audite **le kit repo**, pas le vault. Le standalone `scripts/mem-health-scan.py` (qui scanne des vaults sur des machines pouvant ne pas avoir le kit installé) n'a donc pas d'équivalent. Divergence intentionnelle assumée et documentée dans les en-têtes des deux modules + dans `CLAUDE.md`.

### 5.4 Validation grandeur réelle

Scan ad-hoc sur le kit prod : **96 SKILL.md templates scannés** sur les 4 adapters Anthropic-format, **max description length 962 chars**, **0 finding**. Filet posé, aucun faux positif aujourd'hui.

## 6. Stubs assumés par design (pas par dette)

Deux outils MCP restent intentionnellement skill-only en v0.9.1 :

### `mem_archeo_context` (Phase 1 — extraction de contexte organisationnel/décisionnel/fonctionnel)

La procédure pivote sur la classification sémantique de spans dans 7 catégories (workflow, sync, multi-tenant, security, ADR, fonctionnel, principles). C'est une tâche de classification textuelle nuancée — territoire LLM. Un port Python heuristique produirait des faux positifs/négatifs significatifs et perdrait la valeur sans aucun bénéfice opérationnel (aucun gain de tokens — la classification serait quand même faite par LLM ensuite).

### `mem_archeo_atlassian` (retro-archive Confluence + Jira)

Le **MCP client-side Atlassian** (`mcp__claude_ai_Atlassian__*`) est déjà disponible avec auth gérée (OAuth + API tokens), couverture API riche (Jira issues, Confluence pages, search, attachments) et maintenance assurée par Anthropic. Un port `httpx` maison serait une duplication sans gain : auth à re-implémenter, rate limiting à re-gérer, surface couverte plus pauvre. Décision **réversible** si l'usage révèle un besoin spécifique non couvert par le client-side.

### Visibilité de la décision

Le tableau « Outils MCP disponibles » du README (section Mode MCP) sépare désormais clairement les outils archeo natifs (`mem_archeo`, `mem_archeo_stack`, `mem_archeo_git`) des stubs par design (`mem_archeo_context`, `mem_archeo_atlassian`). La ligne dédiée porte un libellé explicite : **« stubs MCP intentionnels — pas un manque »** avec la justification courte. Évite que la prochaine review interprète les stubs comme un trou.

## 7. Workflow `sync.json` validé en conditions réelles

Le manifest `sync.json` livré en v0.9.0 a été testé en conditions réelles **trois fois** lors du cycle v0.9.x :

1. Patch `mem_archive` enforcement wikilinks (commit `7519db4`) → diff `sync.json` = 1 ligne (hash de `mem_archive`).
2. 10ᵉ catégorie `skill-description-too-long` (commit `7e952d8`) → diff `sync.json` = 1 ligne (hash de `mem_health_scan`).
3. Port Phase 3 niveaux non-tags (cette release v0.9.1) → diff `sync.json` = **0 ligne**. Les procédures `mem-archeo-git.md` et `mem-archeo.md` étaient déjà documentées avec le périmètre cible (4 niveaux + branch-first) ; c'est l'implémentation Python qui rattrapait. Aucun changement de procédure → aucun hash à actualiser. Cas attendu et propre.

Le filet doctrinal fonctionne avec la granularité prévue. Pattern adopté pour usage durable.
