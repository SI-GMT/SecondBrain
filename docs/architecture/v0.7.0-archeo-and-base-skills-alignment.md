# v0.7.0 — Refonte `mem-archeo` triphasée + alignement des skills de base

## 1. Contexte

L'analyse comparative `docs/analyses/2026-04-28-mem-archeo-comparatif-3-llm.md` (Gemini / Codex / Claude Opus sur le même repo Kintsia) a révélé que les divergences inter-LLM ne sont qu'un symptôme. La cause racine : `mem-archeo` ne consulte qu'**une seule fenêtre** (Git) qui ne montre qu'**une dimension** (le temps opérationnel). Sans contextualisation préalable, chaque LLM extrapole différemment ce qui manque.

**Refonte triphasée** :

- **Phase 0 — Topologie repo** : scan structurel partagé (sources, docs, config, tests, CI, infra, manifests, AI files). Bloc inclus `core/procedures/_repo-topology.md` consommé par les 3 phases.
- **Phase 1 — Contextualisation organisationnelle / décisionnelle / fonctionnelle** : lecture des AI files + `docs/` + `cadrage/` + `adr/` + `rfc/` + `CHANGELOG`. Source `archeo-context`. Atomes : principles, goals, knowledge ADR.
- **Phase 2 — Contextualisation technique** : manifestes + conteneurisation + CI/CD + structure dossiers + `.env.example`. Source `archeo-stack`. Atomes : architecture, patterns d'interop, conventions.
- **Phase 3 — Contextualisation temporelle** : la procédure actuelle (Git tags / releases / merges / commits), enrichie par les phases 1 et 2. Source `archeo-git`.

**Architecture skills** : 4 skills au lieu d'1 — `mem-archeo-context`, `mem-archeo-stack`, `mem-archeo-git`, `mem-archeo` (orchestrateur des 3). Sources découplées pour permettre l'invocation individuelle.

**Alignement des skills de base** : les fonctions mem de base doivent reconnaître les nouveaux atomes archeo et la topologie persistée, sinon la refonte produit des artefacts que le reste du système ignore. Trois skills bloquants à patcher (`mem-archive`, `mem-recall`, `_router`) ; quatre souhaitables (`mem-doc`, `mem-rename`, `mem-merge`, `mem-rollback-archive`).

## 2. Topologie persistée — `99-meta/repo-topology/{slug}.md`

### 2.1 Emplacement et nommage

`{VAULT}/99-meta/repo-topology/{project-slug}.md` — un fichier par projet Git ayant fait l'objet d'au moins une `mem-archeo` ou d'un `mem-archive` full-mode (lequel ajoute le snapshot topologique, voir §6.1).

Pas de topologie pour les domaines (un domaine n'a pas de repo Git par définition). Si un domaine apparaît un jour comme adossé à un repo, ce sera un projet déguisé — auquel cas la promotion via `mem-promote-domain` créera l'emplacement projet et la topologie.

### 2.2 Frontmatter

```yaml
---
date: 2026-05-01                   # date du dernier snapshot
zone: meta
type: repo-topology
project: kintsia
repo_path: /Users/ben/Projets/Kintsia    # chemin absolu du repo source
repo_remote: git@github.com:org/kintsia.git    # URL remote si déterminable
content_hash: a3f9c2...             # SHA-256 du corps Markdown — idempotence
previous_topology_hash: 8b1e4d...   # hash de la version précédente, vide si premier snapshot
last_archive: 2026-05-01-14h22-kintsia-v0-3-livre.md    # archive ayant déclenché ce snapshot
tags: [zone/meta, type/repo-topology, project/kintsia]
---
```

`previous_topology_hash` permet le rollback (cf. §6.7) et tracer l'évolution de la topologie au fil des sessions.

### 2.3 Format du corps

```markdown
# Topology — {project-slug}

> Snapshot du repo `{repo_path}` au {date}. Voir aussi : [contexte projet]({path-to-context.md})

## Sources

- `src/` — code applicatif principal
- `lib/` — modules réutilisables

## Documentation

- `README.md`
- `CLAUDE.md`
- `AGENTS.md`
- `docs/`
- `cadrage/`
- `adr/`

## Configuration

- `.env.example`
- `config/`
- `pyproject.toml`

## Tests

- `tests/`
- `__tests__/`

## CI / CD

- `.github/workflows/`
- `azure-pipelines.yml`

## Infrastructure

- `Dockerfile`
- `docker-compose.yml`

## Manifests

- `package.json` — Node.js, deps frontend
- `pyproject.toml` — Python, backend
- `Cargo.toml` — Rust, service auxiliaire

## Stack résolue

- **Frontend** : Next.js 14, React 18
- **Backend** : Python 3.12, FastAPI
- **DB** : PostgreSQL via Supabase self-hosted
- **CI** : GitHub Actions

## Conventions détectées

- Pre-commit hooks (`.pre-commit-config.yaml` détecté)
- ESLint + Prettier (`.eslintrc.json`, `.prettierrc`)
- pytest (`pytest.ini` + `tests/conftest.py`)

## Phases archeo couvertes

- Phase 1 (`archeo-context`) — 4 atomes — dernière passe : 2026-05-01
- Phase 2 (`archeo-stack`) — 7 atomes — dernière passe : 2026-05-01
- Phase 3 (`archeo-git`) — 12 archives — dernière passe : 2026-04-30

## Atomes dérivés des phases archeo

- `[[40-principles/work/architecture/source-of-truth-derived-aggregates]]` — phase 1
- `[[20-knowledge/architecture/auth-proxy-supabase-self-hosted]]` — phase 2
- ...
```

### 2.4 Règles d'évolution

- **Création** : à la première `mem-archeo` (n'importe quelle phase) ou au premier `mem-archive` full-mode sur un projet adossé à un repo détectable.
- **Mise à jour** : `mem-archive` full-mode rescanne et compare via `content_hash`. Si différent, écrase le fichier (le précédent hash est stocké dans `previous_topology_hash` pour rollback).
- **Lecture** : `mem-recall` charge la topologie au début, l'inclut dans le briefing.
- **Suppression** : seulement via `mem-rename` (renommage du projet → renomme le fichier) ou `mem-rollback-archive` avec `--with-topology`.

### 2.5 Détection du repo associé à un projet

Le projet vault `{slug}` peut être adossé à un repo Git via plusieurs heuristiques (en cascade) :

1. Champ `repo_path` dans le frontmatter du `context.md` (à introduire en v0.7.0 — cf. §6.1).
2. Si CWD est un repo Git et son basename `=` `{slug}` ou correspond à un alias documenté → adossement implicite.
3. Si une topologie `{slug}.md` existe déjà → utiliser son `repo_path` comme référence.
4. Sinon : pas de topologie, pas d'archeo automatique, le full-mode `mem-archive` skip silencieusement la phase topologique.

## 3. Frontmatter des atomes archeo

### 3.1 Champs communs aux 3 sources

Tous les atomes produits par `mem-archeo-context`, `mem-archeo-stack`, `mem-archeo-git` portent ces champs en plus du frontmatter universel (cf. `_frontmatter-universal.md`) :

```yaml
source: archeo-context | archeo-stack | archeo-git
content_hash: <sha256>           # SHA-256 du corps de l'atome — idempotence
previous_atom: "[[old-slug]]"    # vide à la première écriture, présent si révision
project: <slug>                  # tag transverse (le projet adossé)
context_origin: "[[archive-or-topology]]"   # archive Git ou topologie qui a fondé l'atome
```

### 3.2 Spécifique `archeo-context` (Phase 1)

```yaml
source: archeo-context
source_doc: <path-relative-to-repo>      # ex: docs/cadrage/01-architecture.md
source_doc_hash: <sha256>                # hash du fichier source au moment de l'extraction
extracted_category: workflow | sync | multi-tenant | security | adr | goal | other
```

**`extracted_category`** correspond aux 5 catégories explicitées dans le correctif 1 de l'analyse 3-LLM, plus `goal` (pour la Phase 1 qui absorbe l'extraction de goals) et `other` pour le reste.

### 3.3 Spécifique `archeo-stack` (Phase 2)

```yaml
source: archeo-stack
source_manifest: <path-relative-to-repo>     # ex: package.json, pyproject.toml
detected_layer: frontend | backend | db | ci | infra | tests | tooling | other
detected_techno: <name-or-list>              # ex: "Next.js 14", ["FastAPI", "Pydantic"]
```

### 3.4 Spécifique `archeo-git` (Phase 3, déjà existant — enrichi)

```yaml
source: archeo-git    # existait déjà
source_milestone: <tag|sha|range>    # existait déjà
commit_sha: <sha>
friction_detected: true | false      # nouveau — cf. correctif 5 (section Friction & Resolution)
```

### 3.5 Idempotence et révision (logique commune R10 du router, étendue)

Avant écriture, le router cherche un atome existant avec **même clé d'idempotence** :

| Source | Clé d'idempotence |
|---|---|
| `archeo-context` | `(project, source_doc, extracted_category)` |
| `archeo-stack` | `(project, source_manifest, detected_layer)` |
| `archeo-git` | `(project, source_milestone, source_atom_type, source_atom_subject)` |
| `archeo-atlassian` | `(confluence_page_id, confluence_updated)` (existant) |

Si trouvé :
- `content_hash` identique → silent skip.
- `content_hash` différent → nouvelle révision avec `previous_atom: [[old]]` + tag `revision`. L'ancien atome reste en place (immuable).

## 4. Règles de collision sémantique côté router

Nouveau bloc R11 dans `_router.md`. Cible : éviter qu'un atome ingéré via `mem-doc`, `mem-note`, `mem-principle` ne crée un doublon d'un atome archeo existant sur le même thème.

### 4.1 Détection

Pour chaque atome candidat (avant écriture), le router calcule :

1. **Slug-subject normalisé** : lowercase, accents stripés, max 60 chars (déjà calculé pour le path).
2. **Recherche dans le vault** d'atomes existants ayant :
   - Même `project` (si le candidat a un projet associé).
   - Même `zone` ET (`type` ou `extracted_category` ou `detected_layer`).
   - Slug-subject **égal** OU **inclus dans** un slug-subject existant (préfixe, ratio Levenshtein > 0.8).

3. Si match trouvé → **collision sémantique candidate**.

### 4.2 Comportements

Selon le source du **candidat** (pas de l'existant) :

| Source candidat | Comportement par défaut | Override flag |
|---|---|---|
| `manual` (mem, mem-note, mem-principle, etc.) | Affiche au user : "Atome similaire détecté: {existing}. Action : merge (M), revision (R), skip (S), force (F)" | `--force-write` ignore la détection |
| `lived` (mem-archive) | Crée quand même (les sessions vécues sont par construction nouvelles), mais signale le chevauchement dans le report | `--strict-collision` bascule en mode interactif comme `manual` |
| `doc` (mem-doc) | Mode interactif systématique : "Le doc {path} chevauche l'atome archeo {existing}. Action: merge dans archeo (M), créer atome doc séparé (S), skip (X)" | `--archeo-context` force la dérivation comme atome archeo-context |
| `archeo-*` | Géré par R10 (idempotence par clé spécifique, pas par collision sémantique) | n/a |

### 4.3 Action `merge`

- Le candidat est rejeté.
- Son contenu est ajouté en append au corps de l'atome existant, dans une section `## Updates` datée.
- Le `content_hash` de l'existant est recalculé.
- Si l'existant est un atome archeo-* : `previous_atom` est mis à jour avec une référence au snapshot précédent ? **Non** — le merge n'est pas une révision, c'est un enrichissement non destructif. Pas de `previous_atom` recalculé.

### 4.4 Action `revision`

- Le candidat devient la nouvelle version.
- L'ancien atome reste en place avec `superseded_by: [[new-slug]]` ajouté dans son frontmatter.
- Le candidat porte `previous_atom: [[old-slug]]`.

### 4.5 Limites connues

- Pas de détection sémantique fine (LLM-based) — c'est un comparateur lexical/structurel. Les paraphrases passent à travers. Acceptable pour v0.7.0.
- Pas de gestion des collisions cross-projet — la détection est scopée au projet.

## 5. Spec delta — skills bloquants v0.7.0

### 5.1 `mem-archive.md` — snapshot topologique en full-mode

**Étape ajoutée** : après l'étape 4 (rewrite context), avant l'étape 5 (update history) :

```
### 4.5. Update repo topology snapshot (if project + repo)

If the target is a project AND a Git repo is associated with it
(see §2.5 — repo_path in context.md frontmatter, CWD match, or pre-existing
topology), then:

1. Invoke {{INCLUDE _repo-topology}} to scan the current state of the repo.
2. Render the topology Markdown body (cf. §2.3 of the architecture doc).
3. Compute content_hash. If equal to the existing topology's hash → skip.
4. Otherwise: read the existing topology, capture its content_hash as
   previous_topology_hash, and atomic-rename-write the new file at
   {VAULT}/99-meta/repo-topology/{slug}.md.
5. Update the new topology's last_archive field to point to the archive
   just created at step 3.

If no repo is associated → skip silently. The full-mode archive is not
blocked by topology absence.
```

**Champs ajoutés au frontmatter du `context.md`** (étape 4) :

```yaml
repo_path: <absolute-path-or-empty>    # nouveau — adossement Git
```

Patch silencieux : si un context.md existant ne porte pas `repo_path`, l'agent essaie de le résoudre (CWD basename match) et l'ajoute. Si aucun match, laisse vide. À la prochaine session sur le projet, le LLM peut demander à l'utilisateur "le projet semble adossé à `~/Projets/{slug}`, est-ce correct ?" pour fixer.

**Frontmatter de l'archive** (étape 3) gagne :

```yaml
topology_snapshot_hash: <sha256-or-empty>    # hash du snapshot pris à cette archive
previous_topology_hash: <sha256-or-empty>    # hash de la topologie avant cette archive
```

Permet à `mem-rollback-archive` de restaurer la topologie précédente (cf. §6.7).

### 5.2 `mem-recall.md` — lecture topologie + atomes archeo

**Étape ajoutée** : entre l'étape 3 (history) et l'étape 4 (attached items) :

```
### 3.5. Load repo topology (new in v0.7.0)

If {VAULT}/99-meta/repo-topology/{slug}.md exists:
- Read it.
- Extract: stack résolue, conventions détectées, comptes par phase archeo.
- Include as a "Project topology" section in the briefing (step 5).

If absent: omit the section. Suggest in the briefing: "Run /mem-archeo
on this project to populate the technical and organizational context."
```

**Étape 4 enrichie** : ajout d'une 4e zone à charger :

```
| 99-meta/repo-topology/ | filename = {slug}.md | Stack and conventions of the project |

Atomes archeo-* attachés au projet sont déjà chargés via les zones
40-principles, 50-goals, 20-knowledge (ils portent project/{slug}). Pas
de zone supplémentaire — c'est le tag qui fait le pont.
```

**Étape 5 enrichie** : nouveau bloc dans le format de réponse :

```
### Project topology

**Stack** : {stack résolue extraite de topology.md}
**Conventions** : {N items}
**Phases archeo** :
  - context: {N atomes}, last pass {date}
  - stack:   {N atomes}, last pass {date}
  - git:     {N archives}, last pass {date}
```

Si la topologie est absente, ce bloc est remplacé par :

```
### Project topology

Not yet captured. Suggest: /mem-archeo to populate.
```

### 5.3 `_router.md` — sources archeo + collisions sémantiques

**R1 (Reception)** mise à jour de l'enum source-hint :

- Source hint (optional): lived | doc | archeo-context | archeo-stack

Ajout de `architecture` aux types `knowledge` pour les atomes Phase 2 type "stack résolue", "pattern d'interop".

## 6. Spec delta — skills souhaitables v0.7.0

### 6.1 `mem-doc.md` — cohabitation avec atomes archeo

**Étape 5 (invoke router)** : passage d'un nouveau hint optionnel `--archeo-context` qui force `source: archeo-context` au lieu de `source: doc`. Le router applique alors la clé d'idempotence archeo-context.

Heuristique de détection auto (Step 1.5 nouveau) :

```
### 1.5. Detect archeo-context candidacy (new in v0.7.0)

If the document path matches one of the heuristics below AND a project is
resolved AND a repo_path is detectable:

- File is in {repo}/docs/, {repo}/cadrage/, {repo}/adr/, {repo}/rfc/.
- Filename matches *cadrage*, *adr-*, *rfc-*, *spec-*.
- Filename = README.md, CLAUDE.md, AGENTS.md, GEMINI.md, MISTRAL.md.

Then ask the user: "This document looks like project context. Ingest as
archeo-context (linked to project topology) or as a regular doc?"

If --archeo-context is passed explicitly, skip the prompt.
```

### 6.2 `mem-rename.md` — atomes archeo + topologie

Ajout d'une étape de renommage :

- Renommer `{VAULT}/99-meta/repo-topology/{old-slug}.md` → `{new-slug}.md`.
- Mettre à jour son frontmatter (`project: {new-slug}`, tags).
- Pour chaque atome archeo-* attaché au projet : update `project` field + tag `project/{new-slug}` + path si le path inclut le slug (rare en archeo, mais possible si on a logé sous `50-goals/work/projects/{slug}/`).
- Update wikilinks pointant vers ces atomes.

### 6.3 `mem-merge.md` — règles fusion topologie

Quand on merge le projet `source` dans `target` :

1. **Topologies** : si les 2 existent → union des sets (sources, docs, manifests, etc.). Conflits sur `repo_path` ou `repo_remote` → demander à l'utilisateur lequel garder.
2. **Atomes archeo source** : retag `project/{target}`, déplacement de path si nécessaire, content_hash inchangé.
3. **Hash de la topologie cible** : recalculé après merge.
4. **Si conflit insolvable** : log dans un fichier `99-meta/merge-conflicts/{timestamp}-{source}-into-{target}.md` pour résolution manuelle.

### 6.4 `mem-rollback-archive.md` — rollback snapshot topo

L'archive cible porte `topology_snapshot_hash` et `previous_topology_hash` (cf. §5.1). Si la topologie courante a un hash qui correspond à `topology_snapshot_hash`, alors :

- Si `previous_topology_hash` est rempli → restaurer la topologie précédente (relit le fichier topologie depuis Git du vault si versionné, sinon affiche un warning : "Cannot restore — vault not Git-tracked").
- Si `previous_topology_hash` est vide → c'était la première topologie ; suppression du fichier topologie.
- Si la topologie courante a évolué entre temps (autre archive depuis) → ne touche pas à la topologie. Affiche un warning.

Flag `--with-topology` requis : par défaut, `mem-rollback-archive` ne touche pas à la topologie pour éviter les rollback en cascade involontaires.

## 7. Workflow v0.7.0 — séquencement

### 7.1 Première archeo d'un projet

```
1. /mem-archeo {repo-path}
2. Phase 0 : scan topologie (en mémoire) — _repo-topology
3. Phase 1 : invoque _router avec source=archeo-context, atomes vers
   40-principles + 50-goals + 20-knowledge
4. Phase 2 : invoque _router avec source=archeo-stack, atomes vers
   20-knowledge (architecture/patterns)
5. Phase 3 : invoque _router avec source=archeo-git, archives vers
   10-episodes/projects/{slug}/archives/, atomes dérivés vers les zones
6. À la fin : écrit la topologie persistée 99-meta/repo-topology/{slug}.md
   avec last_archive = la dernière archive Git créée
```

### 7.2 Cycle session normal

```
Début de session :
  - /mem-recall {slug}
    - lit context.md, history.md, repo-topology/{slug}.md
    - charge atomes archeo-* attachés
    - briefe le LLM avec stature complète

Pendant la session :
  - mem-archive silent mode → met à jour context.md uniquement
  - aucune touche à la topologie

Fin de session :
  - /mem-archive (full mode)
    1. archive lived
    2. derived atoms (principles, goals, knowledge nouveaux)
    3. rewrite context.md
    4. (nouveau) snapshot topologie repo si évoluée
    5. update history.md, index.md
```

### 7.3 Re-archeo d'un projet déjà archeo'é

```
/mem-archeo {repo-path} (re-run)
1. Phase 0 : scan topologie courante
2. Phase 1 : pour chaque doc cible, calcule content_hash, compare aux
   atomes archeo-context existants (clé idempotence). Skip si identique,
   révision si modifié, création si nouveau.
3. Idem Phase 2.
4. Idem Phase 3 (R10 router déjà existant).
5. Topologie : update si content_hash différent.
```

## 8. Migration / rétrocompatibilité

### 8.1 Vaults pré-v0.7.0

Les vaults existants ne portent ni topologie, ni atomes archeo-context/stack. Comportement attendu :

- `mem-recall` : section "Project topology" affiche "Not yet captured" — pas d'erreur.
- `mem-archive` full-mode : si pas de `repo_path` dans le context.md, l'agent demande à l'utilisateur ; si l'utilisateur skip, snapshot topologique skippé.
- `mem-archeo` : tourne et écrit normalement, crée la topologie.

### 8.2 Atomes pré-v0.7.0 produits par l'ancien `mem-archeo`

Tous portent `source: archeo-git`. Restent valides. Pas de migration forcée. Une éventuelle re-archeo Phase 3 avec idempotence skip silencieusement les déjà existants.

### 8.3 Procédure renommée `mem-archeo` → `mem-archeo-git`

Les anciens templates adapters (Claude Code, Gemini, Codex, Vibe) restent fonctionnels jusqu'à la prochaine `deploy.ps1` qui régénère. La régénération supprime les anciens et installe les 4 nouveaux skills.

`mem-archeo` reste invocable — c'est désormais l'orchestrateur.

### 8.4 Script de migration

`scripts/migrate-vault-v06-to-v07.py` (nouveau) — idempotent + dry-run :

1. Pour chaque projet ayant un `context.md` sans `repo_path`, tente une résolution heuristique (CWD candidates dans une liste fournie en paramètre, ou prompt interactif).
2. Optionnellement, pour chaque projet, propose un re-`mem-archeo` Phase 1 + Phase 2 si non encore couvert (les phases sont rapides comparées à Phase 3).

## 10. Risques et points d'attention

- **Boucle de mise à jour topologie ↔ archive** : un `mem-archive` qui met à jour la topologie crée potentiellement un atome qui déclenche un autre `mem-archive`. Garantie : `mem-archive` ne s'auto-invoque pas. La topologie est juste un fichier `99-meta/`, pas une archive.
- **Performance** : scanner un gros monorepo à chaque `mem-archive` full-mode peut être coûteux. Mitigation : la Phase 0 (`_repo-topology`) limite le scan à profondeur 2 par défaut, les exclusions standards (`node_modules/`, `target/`, `dist/`, `.git/`) sont câblées.
- **Multi-LLM idempotence** : tester la nouvelle archeo sur les 4 plateformes pour vérifier que les LLM se conforment aux clés d'idempotence et ne créent pas de doublons. C'est l'objectif visé — mais il faudra le valider empiriquement.
- **Vault non versionné Git** : le rollback de topologie suppose que le vault soit versionné pour récupérer l'état précédent. Sinon le rollback est partiel (suppression seule). Documenter dans le README.
