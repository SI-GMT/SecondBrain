## 1. Contexte et motivation

### 1.1 État actuel (v0.7.5)

Le kit SecondBrain est livré comme un ensemble de **24 skills déclaratifs** (procédures Markdown) dupliqués pour chacun des 5 adapters CLI :

- `adapters/claude-code/skills/*.template.md` (24 fichiers + 24 commands)
- `adapters/codex/skills/*/SKILL.md.template` + `prompts/*.template.md` (24 + 24)
- `adapters/gemini-cli/commands/*.template.toml` (24 fichiers)
- `adapters/mistral-vibe/skills/*/SKILL.md.template` (24 fichiers + bloc instructions)
- `adapters/copilot-cli/skills/*/SKILL.md.template` (24 fichiers + bloc instructions)

Chaque skill **embarque la procédure complète** (résolue à `deploy.ps1` depuis `core/procedures/{nom}.md`) et c'est le **LLM client** qui exécute les opérations filesystem (lire `context.md`, écrire `archive.md`, parser frontmatter, etc.).

Conséquences :

- **Coût en context tokens** : chaque skill chargé représente plusieurs centaines à milliers de tokens dans le system prompt du LLM.
- **Performance hétérogène** : l'opération filesystem effective dépend du modèle. Un modèle rapide peut faire l'opération en 1 tool call, un modèle plus lent peut en faire 5 et se tromper sur l'encodage UTF-8 ou sur la résolution wikilinks.
- **Reproductibilité fragile** : les invariants (encoding, frontmatter universel, résolution wikilinks, atomicité des renames) sont codifiés en prose dans `core/procedures/_*.md` et appliqués au mieux par chaque LLM. Les leçons apprises en v0.7.3.1 (`when-to-script`) et v0.7.4 (`linking discipline`, `archived handling`) ont déjà commencé à factoriser ces opérations dans des scripts Python versionnés (`scripts/mem-historize.py`, `scripts/mem-health-scan.py`) — la v0.8.0 généralise ce pattern.
- **Multiplication des adapters** : passer de 4 à 5 adapters en v0.7.5 a montré que le coût marginal d'une nouvelle CLI est élevé (24 templates à cloner, deploy.ps1 à patcher, conventions à dupliquer). À 6, 7, 8 adapters, ce coût devient prohibitif.

### 1.2 Pourquoi MCP

Le **Model Context Protocol** (MCP) standardise la façon dont les CLI LLM consomment des outils externes via JSON-RPC sur stdio. Les CLI compatibles MCP (Claude Code, Codex, Copilot CLI, Gemini CLI partiellement, et plus à venir) chargent automatiquement un serveur MCP déclaré dans leur config et exposent ses outils au LLM avec leur schéma typé.

Bénéfices pour SecondBrain :

- **Code Python comme source de vérité exécutable** — le serveur Python implémente la logique une fois, garantit les invariants (encoding, frontmatter, atomicité), retourne des résultats structurés (Pydantic models). Le LLM appelle l'outil et reçoit la sortie déjà calculée.
- **Coût en tokens divisé** — la description d'un outil MCP est de l'ordre du paragraphe, vs des centaines/milliers de tokens pour une procédure embarquée.
- **Performance déterministe** — un appel d'outil MCP est une opération atomique (input → résultat structuré), pas une séquence d'opérations filesystem orchestrée par le LLM.
- **Coût marginal d'une nouvelle CLI quasi-nul** — toute CLI compatible MCP charge le même serveur, même config. Plus besoin d'un adapter complet par CLI.
- **Préparation à CollectiveBrain** — un serveur MCP central pourra évoluer vers un serveur partagé d'équipe (Phase 4+).

### 1.3 Pourquoi maintenant

Trois préconditions sont satisfaites :

1. La refonte brain-centric v0.5 a stabilisé le schéma de zones et le router sémantique. Les concepts qu'exposeraient les outils MCP ne bougeront plus structurellement.
2. Les invariants doctrinaux (linking, archived, when-to-script, frontmatter universel) ont été formalisés en blocs `_*.md` factorisables — ils deviennent les briques partagées des modules Python.
3. Le pattern "script Python versionné" est validé en production (v0.7.3.1 mem-health-scan, v0.7.4 mem-historize) — passer du script standalone à l'outil MCP est un saut technique modeste, pas une refonte.

## 2. Décisions structurantes (5 axes)

Tranchées en session de cadrage 2026-05-02 par l'utilisateur (Ben).

### Axe 1 — Granularité des outils

**1 outil MCP par skill** = 24 outils high-level, miroir exact des 24 procédures `core/`. Pas d'outils plus fins (`read_context`, `write_archive`, etc.) — ces primitives sont des fonctions internes du serveur, pas des outils exposés au LLM.

**Justification** : préserve la sémantique des skills (le LLM ne doit pas avoir à composer 5 primitives pour faire un `mem-recall`), simplifie la migration (1-pour-1), et garde un nombre d'outils raisonnable pour la fenêtre de tools du LLM.

### Axe 2 — Transition

**Coexistence MCP-first + skills fallback**. Une CLI dont le MCP `memory-kit` est configuré utilise le MCP en priorité. Sans MCP configuré (ou en cas de défaillance du serveur), elle retombe sur les skills classiques.

**Implication** : les skills déployés dans `~/.claude/skills/`, `~/.codex/skills/`, etc. **restent installés** par `deploy.ps1`. Leur procédure est patchée pour commencer par : "Si l'outil `mcp__memory-kit__mem_X` est disponible, l'invoquer avec ces arguments. Sinon, suivre la procédure ci-dessous."

**Justification** : pas de coupure brutale, supporte les CLI sans MCP (Mistral Vibe à ce jour, futurs entrants), et donne un mécanisme de fallback sain en cas de bug du serveur MCP.

### Axe 3 — Configuration

**5 `memory-kit.json` par CLI** (existants) **+ 1 `~/.memory-kit/config.json` dédié au MCP server**. Total : 6 fichiers synchronisés par `deploy.ps1`.

Le serveur MCP lit **uniquement** `~/.memory-kit/config.json` (override possible via `$MEMORY_KIT_HOME`). Les configs par-CLI restent pour les skills fallback (ils continuent de lire `~/.claude/memory-kit.json`, etc.).

**Justification** : permet au serveur MCP d'être indépendant de toute CLI cliente (utilisable via `pipx install memory-kit-mcp` même sur un poste sans Claude Code), tout en préservant la back-compat des skills.

### Axe 4 — Scope v0.8.0

**24 outils livrés ensemble.** Pas de release "MCP partiel" qui ferait moitié-moitié.

**Justification** : évite l'asymétrie où certains skills sont MCP-first et d'autres skills-only — le mental model utilisateur reste uniforme. La complexité d'écriture des 24 outils est gérable car ils partagent les mêmes primitives (vault config, frontmatter, atomic write, scan).

### Axe 5 — Core comme source de vérité

**Couche d'adaptation Python**, `core/procedures/` reste la source de vérité **fonctionnelle et documentaire**.

Concrètement :
- Les procédures Markdown dans `core/procedures/mem-*.md` continuent de décrire **ce que fait** chaque skill (entrées, sorties, invariants, cas particuliers).
- Le module Python `mcp-server/src/memory_kit_mcp/tools/{nom}.py` est la **traduction exécutable** de cette spec.
- Les skills fallback chargent toujours la procédure résolue (avec `{{INCLUDE _bloc}}`) — ils continuent de fonctionner exactement comme avant.

Discipline de cohérence : tout changement dans une procédure `core/` impose un changement dans le module Python correspondant (et vice-versa). Une catégorie scanner future (`mcp-tool-spec-drift`) pourra détecter les divergences (cible v0.8.x).

**Justification** : préserve la doctrine "single source of truth" (la procédure est toujours documentée au même endroit), permet aux skills fallback de continuer à fonctionner sans dépendance Python, et fait du module Python une implémentation lisible plutôt qu'une duplication de la spec.

## 3. Stack technique

| Composant | Choix | Justification |
|---|---|---|
| **Framework MCP** | `fastmcp` ≥2.13 (jlowin/fastmcp standalone) | Décorateur-based, type-hints + Pydantic, **Client in-memory pour tests pytest sans subprocess** (gain majeur pour atteindre 80% coverage). Le `mcp.server.fastmcp` officiel est aussi viable mais sans Client in-memory. |
| **Build backend** | hatchling | Standard moderne, recommandé par les quickstarts MCP officiels. |
| **Env management** | uv | Rapide, déterministe, pattern PEP 723 cohérent avec les scripts existants. |
| **Python** | ≥3.12 | Syntax `T \| None`, performance, support typing moderne. |
| **Validation** | pydantic ≥2.9 | Schémas typés pour args et retours, sérialisation auto en `structuredContent` MCP. |
| **Distribution** | pipx install | Isolation, console script `memory-kit-mcp` sur le PATH, met à jour facilement. |
| **Tests** | pytest ≥8 + pytest-asyncio + pytest-cov | `--cov-fail-under=80` enforced. Client in-memory pour invoquer les outils sans I/O. |
| **Lint** | ruff ≥0.6 | Rapide, déjà standard dans l'écosystème Python. |
| **Transport** | stdio uniquement | Lance par le client CLI à chaque session, pas de serveur HTTP à gérer. |

### Versionnage du paquet

`memory-kit-mcp` suit la version du kit (v0.8.0 = première release MCP). Tag git unique du kit, pas de versionnage indépendant — quand le kit passe à v0.9.0, le MCP server passe à v0.9.0 aussi.

## 6. Schémas d'arguments et de retour

### Convention générale

Tous les outils utilisent **type hints Python 3.12+** (syntax `T | None`) et **Pydantic v2** pour les schémas riches. Trois patterns selon la complexité :

```python
# (a) Hints simples — outils sans args ou avec args triviaux
@mcp.tool()
def mem_list() -> ListResult:
    """List all projects and domains in the vault."""
    ...

# (b) Pydantic Field pour rich schema — args avec contraintes
from pydantic import Field
@mcp.tool()
def mem_search(
    query: str = Field(..., description="Full-text search query"),
    scope: str = Field("all", pattern="^(all|projects|domains|knowledge|principles|goals|people)$"),
    include_archived: bool = False,
    limit: int = Field(50, ge=1, le=500),
) -> SearchResult:
    """Search vault content. Returns ranked matches with context snippets."""
    ...

# (c) Pydantic BaseModel pour args groupés / nested — outils complexes
from pydantic import BaseModel
class ArcheoOpts(BaseModel):
    repo_path: str
    branch_first: str | None = None
    branch_base: str = "main"
    by_author: bool = True
    allow_archived: bool = False

@mcp.tool()
def mem_archeo(opts: ArcheoOpts) -> ArcheoResult:
    """Triphasic archeo orchestrator. ..."""
    ...
```

### Retour : Pydantic BaseModel par défaut

Chaque outil retourne un `BaseModel` typé. FastMCP sérialise automatiquement en `structuredContent` (consommable par le LLM via JSON Schema) **et** en `text` (rendu lisible par défaut). Le LLM voit les deux.

```python
class RecallResult(BaseModel):
    project: str
    kind: str                       # "project" | "domain"
    last_session: str               # ISO date
    phase: str
    scope: str
    state: dict[str, list[str]]     # {"validated": [...], "in_progress": [...]}
    decisions: list[str]
    next_steps: list[str]
    active_principles: list[LinkRef]
    open_goals: list[LinkRef]
    architecture_atoms: list[LinkRef]
    key_people: list[PersonRef]
    assets: list[str]
    archived: bool = False          # True if loaded from 10-episodes/archived/
    briefing_md: str                # Markdown rendering (already formatted)
```

Les outils qui modifient le vault (mem_archive, mem_rename, mem_merge, etc.) retournent un `ChangeReport` standardisé :

```python
class ChangeReport(BaseModel):
    skill: str                       # "mem_archive"
    success: bool
    files_created: list[str]
    files_modified: list[str]
    files_deleted: list[str]
    files_moved: list[tuple[str, str]]
    warnings: list[str]
    summary_md: str                  # human-readable summary
```

### Localisation des messages

Les chaînes user-facing (prefixe section, libellés, messages d'erreur) sont résolues via `core/i18n/strings.yaml` (déjà existant) à l'invocation de chaque outil. Le serveur lit la langue depuis `~/.memory-kit/config.json` champ `language`.

### Codes d'erreur

Les erreurs métier (project not found, vault unreachable, archived without revive flag, etc.) lèvent `mcp.shared.exceptions.McpError` avec un `code` standardisé :

```python
INVALID_PARAMS = -32602        # bad args
PROJECT_NOT_FOUND = -32001     # custom range -32000 to -32099
VAULT_UNREACHABLE = -32002
ARCHIVED_REQUIRES_OVERRIDE = -32003
WIKILINK_DANGLING = -32004
FRONTMATTER_INVALID = -32005
HASH_MISMATCH_CONCURRENCY = -32006
```

Le LLM client traduit ces codes en réaction utilisateur (retry, demande de clarification, message d'erreur).

## 7. Couche d'adaptation : `core/procedures/` comme contrat

### Principe de double-emploi

Chaque procédure `core/procedures/mem-X.md` joue **deux rôles simultanés** :

1. **Spec fonctionnelle** lue par les humains et par les LLM en mode skills fallback. C'est le contrat : ce que fait le skill, ses cas particuliers, ses invariants.
2. **Référence d'implémentation** pour le module Python `mcp-server/src/memory_kit_mcp/tools/X.py`. Le module Python implémente exactement ce que la procédure décrit.

### Discipline de cohérence

- Modifier `core/procedures/mem-X.md` **et** `tools/X.py` dans le **même commit**. Une PR qui change l'un sans l'autre est suspecte.
- Les invariants partagés (encoding, frontmatter universel, linking, archived handling) restent dans les blocs `core/procedures/_*.md`. Le module Python `vault/` les implémente comme primitives ; les modules `tools/` les invoquent via les primitives.
- Pas de duplication de la spec dans le code Python. Les commentaires `# Voir core/procedures/mem-recall.md §3` pointent vers la procédure source.

### Sub-skill : pattern de pointage explicite

En tête de chaque fichier `tools/X.py`, un docstring de module pointe vers la procédure source :

```python
"""mem_recall — Load a project's context from the vault.

Spec: core/procedures/mem-recall.md
Shared blocks: _router.md, _frontmatter-universal.md, _linking.md, _archived.md
"""
```

### Détection de drift (futur)

Catégorie scanner `mcp-tool-spec-drift` à ajouter à `mem-health-scan` v0.8.x : compare le hash de la procédure à un `last_synced_hash` dans le frontmatter du module Python (ou un fichier `mcp-server/sync.json`). Si drift, lever un warning. Ne bloque pas, signale.

```
~/.claude/memory-kit.json           # existant — lu par les skills Claude Code (fallback)
```


**Codex** (`~/.codex/config.toml`) :
```toml

```

**Copilot CLI** (`~/.copilot/mcp-config.json`) :
```json
{
  "mcpServers": {
    "memory-kit": {
      "command": "memory-kit-mcp",
      "args": []
    }

```

**Gemini CLI** : à confirmer pendant l'implémentation. Si Gemini CLI v0.8 ne supporte pas encore MCP, l'extension `memory-kit` continue d'être déployée (skills-only).

**Mistral Vibe** : pas de support MCP à ce jour. Skills-only.

### Cohabitation skills + MCP

Sur une CLI compatible MCP qui a aussi les skills installés, le MCP-first prévaut (le skill commence par "if MCP available, delegate"). Si l'utilisateur supprime explicitement le MCP server, les skills prennent le relais sans changement.

### Pré-requis runtime

Le serveur Python est invoqué par le binaire `memory-kit-mcp` installé via `pipx install ./mcp-server`. `deploy.ps1` :
1. Détecte la présence de `pipx` (recommandé). Sinon, propose `python -m pip install --user ./mcp-server` en fallback.
2. Lance `pipx install ./mcp-server --force` (ou l'équivalent pip) pour installer/mettre à jour le serveur.
3. Vérifie que `memory-kit-mcp` est sur le PATH (sinon affiche un avertissement).
4. Écrit `~/.memory-kit/config.json` + injecte la config MCP dans chaque CLI compatible.

## 9. Pattern MCP-first / skills-fallback

### Pattern dans la procédure du skill

Chaque procédure `core/procedures/mem-X.md` gagne un **bloc d'introduction** (au-dessus du contenu actuel) :

```markdown
## Mode d'exécution

**MCP-first** : si l'outil `mcp__memory-kit__mem_X` est disponible dans la session, l'invoquer avec les arguments correspondants (cf. § Arguments). Renvoyer le résultat tel quel à l'utilisateur. Ne pas exécuter les opérations filesystem ci-dessous.

**Fallback skills-mode** : si l'outil MCP n'est pas disponible (CLI sans MCP configuré, ou serveur MCP indisponible), exécuter la procédure complète ci-dessous.
```

Ce bloc est **inséré automatiquement** par `deploy.ps1` au-dessus du contenu actuel de chaque procédure résolue. Pas de duplication dans `core/` — c'est un bloc `_mcp-first.md` injecté comme `_frontmatter-universal.md` aujourd'hui.

### Implémentation côté deploy.ps1

Modification du `Resolve-IncludeDirectives` (ou ajout d'un step post-include) qui prepend le bloc `_mcp-first.md` (en substituant `{{TOOL_NAME}}` par `mem_X` dérivé du nom de fichier) avant la résolution finale.

### Détection MCP côté LLM

C'est au LLM de décider, à l'invocation du skill, si l'outil MCP est disponible. Tous les LLM modernes (Claude, GPT-4+, Gemini) listent leurs outils disponibles dans leur prompt. Le bloc d'introduction du skill rend la décision triviale ("si tu vois `mcp__memory-kit__mem_X` dans tes outils, appelle-le").

### Risque : divergence comportement

Si la procédure markdown évolue mais pas le module Python (ou inverse), MCP-first et skills-fallback peuvent diverger. Mitigation : la discipline de cohérence (§7) + la catégorie scanner future `mcp-tool-spec-drift`.

## 12. Risques et mitigations

| Risque | Impact | Mitigation |
|---|---|---|
| **Drift core/procedures vs tools/Python** | Skills fallback et MCP divergent silencieusement. Bug invisible jusqu'à un cas d'usage qui révèle l'écart. | Discipline de cohérence (§7) + catégorie scanner future `mcp-tool-spec-drift` (v0.8.x). Pour la v0.8.0 : checklist en revue de PR. |
| **Performance du serveur stdio** | Cold start coûteux (init Python + fastmcp). Premier appel d'outil long. | Acceptable pour un usage CLI interactif. Si problème confirmé : revoir l'init paresseuse des modules `tools/` (lazy import). |
| **Compatibilité fastmcp standalone vs SDK officiel** | Fastmcp 2.x diverge de la spec MCP officielle dans certains détails. Risque de rupture future. | Acceptation du risque pour le gain en testabilité. Si rupture : la migration vers `mcp.server.fastmcp` officiel reste mécanique (changer l'import + perdre le `Client` in-memory pour les tests). |
| **pipx absent sur le poste cible** | Install échoue, l'utilisateur reste en skills-only sans s'en rendre compte. | `deploy.ps1` détecte pipx et propose `pip install --user` en fallback. Affiche un avertissement clair. |
| **Vault path Windows avec espaces ou caractères spéciaux** | Crashes au passage de chemin via JSON-RPC ou subprocess. | Tests unitaires sur fixtures avec espaces dans le chemin + chars unicode. |
| **Concurrence MCP server multi-instance** | Si 2 CLI ouvertes utilisent le MCP en parallèle, conflit possible sur les écritures vault. | Pattern atomic write existant (`atomic_io.py`) + hash check read-before-write (cf. `_concurrence.md` doctrinal). À tester explicitement. |
| **Refonte adapters lourds en chunk 11** | `deploy.ps1` patch complexe, risque de régression sur les CLI sans MCP. | Tests manuels sur les 5 CLI avant merge. Le bloc `_mcp-first.md` ajoute juste un prefix conditionnel — si la condition échoue le LLM exécute la procédure existante, donc rétrocompatible par construction. |
| **Tests pytest 80% irréaliste sur certains tools** | Outils complexes (mem_archeo, mem) avec branches multiples — atteindre 80% peut nécessiter des fixtures massives. | Accepter une coverage par module variable (60-95%) tant que la moyenne pondérée dépasse 80%. Tests E2E manuels compensent. |

## 13. Hors scope (post-v0.8.0)

- **Serveur MCP HTTP/SSE** — pour usage distant ou multi-utilisateur (CollectiveBrain). Pour l'instant stdio-only.
- **Cache de session côté MCP** — si un même outil est appelé 5 fois dans la session, on relit le filesystem 5 fois. Acceptable pour la v0.8.0.
- **Refonte i18n strings.yaml en module Python natif** — pour l'instant le serveur lit le YAML existant via `kit_repo`. Migration possible v0.9+.
- **Outil `mem-purge` (suppression définitive d'un projet archived)** — déjà mentionné en cadrage v0.7.4, à scoper v0.9+.
- **Tests E2E automatisés sur les 5 CLI** — utilise les CLI réelles dans des sandboxes (Docker ?). Lourd, possible v0.9+.
- **MCP Resources et Prompts** (autres primitives MCP) — uniquement Tools en v0.8.0. Resources (lecture seule) et Prompts (templates) seraient utiles mais sortent du scope.
- **CollectiveBrain MCP server distribué** — Phase 4+, dépend de l'infra collective.
