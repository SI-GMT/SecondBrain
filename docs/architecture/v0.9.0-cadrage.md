## 1. Contexte et continuité v0.8.0 → v0.9.0

La v0.8.0 a livré le serveur MCP `secondbrain-memory-kit` avec 24 outils déclarés (`mem_*`), dont **19 fonctionnels** (cycle session, inventaire, gestion vault, hygiène 4 catégories de base, ingestion 6 outils) et **5 stubs `mem_archeo*`** qui surfaçaient un fallback explicite vers les skills.

La v0.9.0 boucle ce périmètre :
- Termine le port archeo (Phases 0, 2, 3 + orchestrator) tout en actant que Phase 1 (`mem_archeo_context`) et `mem_archeo_atlassian` restent skills-only par décision design.
- Étend `mem_health_scan` aux 9 catégories canoniques (les 4 restantes étaient en stubs) via une lib partagée.
- Intègre les 6 doc-readers PDF/DOCX/PPTX/XLSX/CSV/HTML directement au serveur MCP (auparavant invoqués via `uv run` côté skills).
- Introduit un filet doctrinal contre la dérive silencieuse entre `core/procedures/` et les modules Python : le manifest `sync.json` + une 9ᵉ catégorie health-scan.

À l'arrivée : **189 tests pytest, 84 % coverage** (vs 114 / 94 % en v0.8.0 — la baisse de coverage est mécanique, le périmètre porté ayant doublé).

## 2. Décisions structurantes (4 axes)

### Axe 1 — Stub par design, pas par dette

Phase 1 (`mem_archeo_context`) et `mem_archeo_atlassian` ne seront pas portés en code Python. Justification :

- **Phase 1** : la procédure pivote sur la classification sémantique de spans en 7 catégories (organisationnel, décisionnel, fonctionnel, etc.). C'est du LLM, pas du Python heuristique. Documenté inline dans `archeo_context.py`.
- **Atlassian** : le client-side MCP Atlassian (`mcp__claude_ai_Atlassian__*`) est déjà disponible avec auth gérée et plus riche qu'un port `httpx` maison. Port natif = duplication sans bénéfice utilisateur. Décision actée et documentée.

Ce choix consolide le principe **« on laisse au LLM ce qui est mieux fait côté LLM »** — le serveur MCP n'est pas une fin en soi, c'est un outil quand il apporte un gain (déterminisme, performance, économie de tokens).

### Axe 2 — Composition orchestrator sans appels MCP imbriqués

`mem_archeo` est un orchestrateur qui chaîne `mem_archeo_stack` puis `mem_archeo_git` en partageant un Phase 0 unique. Pour permettre cette composition sans appels MCP imbriqués (qui passeraient par stdio + sérialisation), la logique métier de chaque tool est extraite dans une fonction module-level `execute_X(...)` et le décorateur `@mcp.tool()` n'enveloppe qu'un thin wrapper.

```python
def execute_stack(repo_path: Path, ...) -> ArcheoStackResult:
    # ... vraie logique
    ...

def register(mcp):
    @mcp.tool()
    def mem_archeo_stack(repo_path: str, ...) -> ArcheoStackResult:
        return execute_stack(Path(repo_path), ...)
```

Pattern à dupliquer pour tout futur outil composable.

### Axe 3 — Lib partagée vs duplication standalone

Trois paires de cohérence existent désormais par construction (cf. section 7) :

1. `core/procedures/mem-X.md` ↔ `mcp-server/src/memory_kit_mcp/tools/X.py`
2. `deploy.ps1` ↔ `deploy.sh`
3. `scripts/mem-health-scan.py` ↔ `memory_kit_mcp.health.scan`

La v0.9.0 ajoute :

4. `scripts/doc-readers/*.py` ↔ `memory_kit_mcp.readers.*`

Le pattern est uniforme : deux endroits par construction, justifiés par autonomie standalone (utilisateurs en mode skills sans MCP installé) ou plateforme cible. Tout changement co-commité, source canonique désignée.

### Axe 4 — Filet doctrinal contre la dérive silencieuse

Le pattern « source de vérité fonctionnelle dans `core/procedures/`, implémentation Python dans `tools/` » repose sur la discipline du développeur pour rester aligné. Une procédure modifiée sans toucher au module Python (ou l'inverse) crée une divergence silencieuse jamais détectée.

Le **manifest `sync.json`** trace le SHA-256 du body de chaque procédure au dernier moment où le développeur a manuellement reconnu que core et Python étaient alignés. La 9ᵉ catégorie health-scan `mcp-tool-spec-drift` compare le hash courant à celui du manifest. Si différent → finding `info` non bloquant qui surface la dérive et incite à réviser le module Python correspondant, puis à lancer `python -m memory_kit_mcp.sync update` pour réécrire le manifest.

## 4. Bloc 2 — Audit hygiène 9 catégories complet

### 4.1 Lib partagée `memory_kit_mcp.health.scan`

Refactor du POC v0.8.0 (4 catégories) vers une lib complète qui implémente les 9 catégories canoniques :

- `malformed-frontmatter` (error)
- `stray-zone-md` (warning)
- `empty-md-at-root` (warning)
- `missing-zone-index` (warning)
- `missing-display` (info, auto-fixable)
- `dangling-wikilinks` (info)
- `orphan-atoms` (warning)
- `missing-archeo-hashes` (warning)
- `mcp-tool-spec-drift` (info, mcp-only — cf. Bloc 4)

`tools/health_scan.py` devient un thin wrapper qui appelle `scan_vault(vault, only_filter=category, kit_repo=config.kit_repo)`.

### 4.2 Discipline de cohérence

`scripts/mem-health-scan.py` (script versionné autonome) garde sa propre copie des 8 premières catégories — règle établie en v0.7.3.1 (`_when-to-script.md`). La 9ᵉ catégorie est mcp-only par design : elle audite le kit repo, pas le vault, donc le standalone n'a pas d'équivalent. Divergence intentionnelle documentée dans les en-têtes des deux modules + dans `CLAUDE.md`.

## 5. Bloc 3 — Doc-readers vendorisés

### 5.1 Avant vs après

| Élément              | Avant (v0.8.0)                                  | Après (v0.9.0)                                  |
|----------------------|-------------------------------------------------|-------------------------------------------------|
| 6 readers (pdf/...)  | `scripts/doc-readers/*.py` invoqués via `uv run` côté procédure skills | `memory_kit_mcp.readers/` package interne, dispatcher `read_document` |
| Activation côté MCP  | `tools/doc.py` raise `NotImplementedError` pour tout sauf md/txt | Fast-path natif md/txt + dispatch sur tous les autres formats |
| Deps                 | Auto-résolues par `uv` PEP 723 inline metadata  | Extra `[doc-readers]` dans `pyproject.toml`     |

### 5.2 API uniforme

Tous les readers exposent `extract(path: Path) -> tuple[str, list[str]]` (markdown, warnings). Les imports lourds (`pypdf`, `python-docx`, `python-pptx`, `openpyxl`, `bs4`, `lxml`) sont **lazy** dans chaque module. Si la dep manque, raise `DocReaderDependencyError` avec hint `pip install memory-kit-mcp[doc-readers]`.

### 5.3 Cas PDF scanné

Si le reader PDF parse le fichier mais yield moins de 100 caractères (seuil `SCAN_THRESHOLD_CHARS`), il return `("", warnings)`. `tools/doc.py` raise alors une `ValueError` avec hint « fall back to native LLM reading » — le LLM client peut intercepter et basculer sur sa lecture vision native. Comportement identique au standalone (exit code 2) traduit dans le contexte MCP.

### 5.4 Discipline de cohérence (4ᵉ paire)

Les heuristiques (seuil scan, MAX_ROWS/MAX_COLS, NOISE_TAGS, fallback encodage utf-8-sig→utf-8→cp1252→latin-1) sont alignées à l'identique entre les deux endroits. Tout changement co-commité ; source canonique = scripts standalone.

## 6. Bloc 4 — Spec-drift scanner et manifest sync.json

### 6.1 Manifest

`mcp-server/src/memory_kit_mcp/sync.json` (versionné, livré dans le wheel) contient pour chaque outil MCP :

```json
{
  "tools": {
    "mem_recall": {
      "procedure": "mem-recall.md",
      "last_synced_hash": "4e1d83ac604ee8bc44b36d6494119738e3683eeb278cbabd4301b34242c552f3"
    },
    ...
  }
}
```

24 entrées (1 par tool), maintenues à la main via le CLI :

```
python -m memory_kit_mcp.sync update --kit-repo C:\_PROJETS\DEVOPS\SecondBrain
```

### 6.2 Calcul du hash

`compute_procedure_hash` :

1. Lit le fichier en `utf-8-sig` (consomme un éventuel BOM).
2. Normalise les fins de ligne CRLF/CR → LF.
3. Strip le frontmatter YAML s'il y en a un (les évolutions cosmétiques de la `description` SKILL.md ne doivent pas trigger drift).
4. SHA-256 du body restant.

### 6.3 Scanner

La 9ᵉ catégorie `mcp-tool-spec-drift` (severity `info`, jamais bloquante) :

- Skip silencieux si `kit_repo` non configuré ou `sync.json` absent (déploiements installés via `pipx` sans clone du repo).
- Sinon, pour chaque entrée du manifest, recalcule le hash courant et compare.
- Si différent → finding « Procedure has drifted since last sync — review the `mem_X` implementation in tools/ and run `python -m memory_kit_mcp.sync update` after reconciling. »
- Si le fichier procédure référencé a disparu → finding « Procedure file missing for tool `mem_X`. »

### 6.4 Workflow développeur

Lors de toute modification core ↔ Python :

1. Modifier `core/procedures/mem-X.md` ET `mcp-server/src/memory_kit_mcp/tools/X.py` ensemble (même commit).
2. Lancer `python -m memory_kit_mcp.sync update --kit-repo ...` pour recalculer les hashes.
3. Inclure `sync.json` dans le même commit.

Le scanner ne flaggera pas tant que `sync.json` est aligné. Si le développeur oublie l'étape 2, la prochaine `mem_health_scan` surfacera la dérive.

## 7. Disciplines de cohérence formalisées

Toutes documentées dans `CLAUDE.md` du repo :

| # | Paire                                                                | Source canonique         | Documenté depuis |
|---|----------------------------------------------------------------------|--------------------------|------------------|
| 1 | `core/procedures/mem-X.md` ↔ `mcp-server/src/memory_kit_mcp/tools/X.py` | core (fonctionnel)       | v0.8.0           |
| 2 | `deploy.ps1` ↔ `deploy.sh`                                           | parité                   | v0.8.x           |
| 3 | `scripts/mem-health-scan.py` ↔ `memory_kit_mcp.health.scan`          | script versionné         | v0.9.0 (Bloc 2)  |
| 4 | `scripts/doc-readers/*.py` ↔ `memory_kit_mcp.readers.*`              | scripts versionnés       | v0.9.0 (Bloc 3)  |

Pattern unique : deux endroits par construction (justifié par plateforme cible ou autonomie standalone), tout changement co-commité, source canonique désignée.

Le manifest `sync.json` (Bloc 4) est l'instrument qui rend la paire #1 mesurable et auditable. Les paires #2, #3, #4 restent vérifiées par discipline du développeur (pas de manifest équivalent — pas de besoin identifié).

## 8. Hors scope (post-v0.9.0)

Renvoyé v0.9.x ou v1.0 :

- **Phase 3 archeo niveaux non-tags** (`releases`, `merges`, `commits`) + mode `--branch-first` → port mécanique progressif, v0.9.x si demandé.
- **Phase 1 archeo (`mem_archeo_context`)** → port mécanique partiel possible (extraction des sections AI files, README, docs/, cadrage/, adr/) en gardant la classification 7-catégories LLM-driven. v0.9.x si demandé.
- **Atlassian port REST** → décidé skill-only par design, pas dans la roadmap.
- **E2E 7 cibles MCP (sandbox Docker)** : le serveur est testé in-memory via `fastmcp.Client`. Une CI matrix qui charge effectivement le serveur dans Claude Code, Codex, Copilot, Vibe, Gemini, Claude Desktop, Codex Desktop reste à monter. Cible v1.0.
- **Catégorie scanner `skill-description-too-long`** (validation `description` SKILL.md ≤ 1024 chars) — durcissement doctrinal v0.7.5, cible v0.9.x.
- **Catégorie scanner `dangling-wikilinks` avec frontmatter `dangling_intentional: ["X"]`** pour exempter les wikilinks intentionnellement non-résolus. Cible v0.9.x.
- **Renforcement `mem-archive` pour vérifier résolution wikilinks à l'écriture** : mécanisme de résolution différée nécessaire pour wikilinks vers atomes en cours de création.
