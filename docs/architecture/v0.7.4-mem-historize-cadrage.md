# v0.7.4 — `mem-historize` (cadrage, non implémenté)

> Cadrage acté en session v0.7.3, à livrer dans une release ultérieure (cible **v0.7.4**, mais peut glisser derrière v0.8.0 MCP si la priorité change).

## Problème résolu

Au fil des sessions, le vault accumule des projets terminés (`phase: archived` ou équivalent dans l'esprit). Ces projets restent valides historiquement mais alourdissent à chaque démarrage :

- `mem-recall` les compte dans le briefing global.
- `mem-list` les affiche au même rang que les projets actifs.
- `mem-search` les scanne par défaut.
- L'index racine les liste tous, ce qui dilue la lisibilité.

Conséquence : la consommation de tokens du LLM au démarrage croît linéairement avec le nombre de projets, **dont la majorité est inactive**. À l'échelle d'un second cerveau utilisé sur plusieurs années, ce coût devient le facteur dominant.

## Concept

`mem-historize` déplace un projet déclaré "fini" vers une zone d'accès second-rang, **toujours présente dans le vault** mais **exclue par défaut des opérations d'accès chaud**.

## Décisions prises

### Localisation : `10-episodes/archived/`

On reste dans `10-episodes/` (mémoire épisodique) — ce sont toujours des projets, pas des knowledge items. Sous-dossier `archived/` à côté de `projects/` et `domains/`. Pas de nouvelle zone racine `90-archived/` : éviter de fragmenter le schéma 9-zones, l'archivage est un **statut**, pas une **classe ontologique**.

```
10-episodes/
├── projects/        # actifs
├── domains/         # actifs (transverses, permanents)
└── archived/        # historisés (anciens projets)
    ├── codemagdns/
    │   ├── context.md       # frozen au moment de l'historisation
    │   ├── history.md
    │   └── archives/...
    └── ...
```

### Critère d'historisation : déclaratif explicite

Pas d'auto-détection (par silence > N jours, par phase contenant "done", etc.). L'utilisateur **dit** quand archiver — `/mem-historize {projet}` ou phrase naturelle "archive le projet codemagdns, c'est fini".

Le frontmatter du `context.md` du projet historisé est patché avec `phase: archived` + `archived_at: {date}`. Idempotent : ré-historiser un projet déjà archivé = noop avec message.

### Réversibilité

`/mem-historize {projet} --revive` (ou "ressuscite codemagdns" en naturel) rapatrie le projet dans `10-episodes/projects/`, retire `phase: archived`, met à jour les liens d'index.

### Comportement des skills d'accès

| Skill | Comportement par défaut sur archived | Override |
|---|---|---|
| `mem-recall` | Refuse silencieusement de charger un projet archived sauf demande explicite par slug. Le briefing post-recall ne mentionne pas la liste des projets archived (compte agrégé seulement). | `mem-recall {slug}` charge même un archived. |
| `mem-list` | Section `### Archived ({N})` au bas de l'inventaire, repliée par défaut (juste le compte). | `--include-archived` déplie la liste complète. `--archived-only` n'affiche que les archived. |
| `mem-search` | Exclut `10-episodes/archived/**` du scan. | `--include-archived` réinclut. `--archived-only` recherche uniquement dans archived. |
| `mem-digest` | Refuse sur un slug archived sauf override. | `--from-archived` autorise. |
| `mem-archive` | Plus aucune écriture vers un projet archived (lever erreur "project archived, /mem-historize --revive d'abord"). | Pas d'override — la règle est dure. |
| `mem-rename` / `mem-merge` | Opèrent sur archived comme sur les autres. Pas de restriction. | — |
| `mem-rollback-archive` | Idem. | — |
| `mem-archeo*` | Refuse par défaut (un repo archeo'd doit être actif). | `--allow-archived`. |

### Index et frontmatter

Le `index.md` racine ajoute une section `## Archived projects` séparée (pas dans `## Projects`). Format :
```
## Archived projects ({N})

- [{slug}](10-episodes/archived/{slug}/history.md) — archived: {date}
```

Le `display` du `context.md` archivé suit la convention `"{slug} — context [archived]"` pour signaler visuellement dans le graph view.

### CollectiveBrain et `collective: true`

L'historisation **n'a pas d'effet sur le flag `collective`**. Un atome promu `collective: true` reste promu, même si son projet est archived. La promotion est un état orthogonal au cycle de vie projet.

### Règle de "no-zombie"

Si un projet est historisé puis le repo associé (`repo_path` du frontmatter) est supprimé, le projet reste dans `archived/` — il n'est **jamais** automatiquement effacé. La suppression définitive d'un projet archived passe par un autre skill futur (cadré en v0.8+ : `mem-purge {slug}` avec confirmation triple, hors scope ici).

## Implémentation pressentie (non engagée)

- Skill `mem-historize` + `mem-historize-revive` (ou flag `--revive` sur le même).
- Patch des skills d'accès listés plus haut pour respecter le statut archived (la majorité doit juste filtrer `archived/` du scan ; le coût d'implémentation est faible et bien réparti).
- Migration script `scripts/migrate-vault-add-archived-zone.py` pour créer `10-episodes/archived/` si absent (idempotent, dry-run par défaut).
- Test bout-en-bout sur un projet réel du vault GMT (probablement `codemagdns` qui est déjà en phase de stabilité).

## Risques et points ouverts

- **Référencement croisé** : si un atome transverse (knowledge, principle) cite un projet archived via `project: {slug}`, faut-il le réétiqueter ? Décision actée : **non**. Le `project:` reste valide (le projet existe toujours, juste dans `archived/`). Les skills qui résolvent `project: {slug}` cherchent dans `10-episodes/projects/{slug}/` puis `10-episodes/archived/{slug}/`.
- **Coût en tokens du briefing** : à vérifier empiriquement. L'hypothèse est que la majorité du gain vient de **ne pas charger** les `context.md` archived. Mesurer avant/après sur le vault GMT.
- **Faux positifs `unlinked-atom`** : `mem-health-scan` doit traiter les wikilinks vers `10-episodes/archived/{slug}/...` comme valides. À ajouter dans le scan v0.7.4 si nécessaire.

## Lien à v0.7.3

v0.7.3 livre `mem-health-scan` / `mem-health-repair` qui forment la **fondation** sur laquelle `mem-historize` s'appuiera : un projet archived peut accumuler des défauts (atomes orphelins, hashes manquants) qu'on veut nettoyer **avant** historisation. Workflow type :

```
/mem-health-scan
/mem-health-repair --apply
/mem-historize {projet}
```

L'ordre garantit qu'on n'archive pas un projet "sale".
